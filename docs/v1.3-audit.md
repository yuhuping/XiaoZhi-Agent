# XiaoZhi v1.3 审计

## 审计范围

- 审计对象是当前工作区中的 v1.3 代码实现，而不是某个干净的 Git tag。
- 当前运行版本来自 `app/main.py`，版本号为 `1.3.0`。
- 本文只保留和 v1.4 下一步改造直接相关的内容，重点聚焦：
  - 当前核心链路
  - `SSE` 流式输出到前端
  - 图片输入保持原图透传
  - 与上述两项直接相关的测试

## 1. 当前核心链路

### 1.1 入口

- FastAPI 入口在 `app/main.py`，应用启动时通过 `lifespan` 初始化 `chat_service`。
- 主接口是 `POST /api/v1/chat/explain-and-ask`，路由位于 `app/api/chat.py`，返回类型固定为 `ChatResponse`。
- 前端 Playground 在 `app/frontend/index.html`，当前通过 `fetch` 请求 JSON 接口，收到完整响应后一次性渲染，没有流式协议。

### 1.2 Service 装配

- `app/services/chat_service.py` 负责组装整条链路：
  - `ModelService`
  - `MemoryManager` 和 `MemoryTool`
  - `LocalKnowledgeRetriever`
  - `BasicTools`
  - `AgentGraph`
- `ChatService.explain_and_ask()` 的主流程是：
  - `build_initial_state(request)`
  - 注入 `state["rag_enabled"]`
  - `await self.graph.run(state)`
  - 从 `final_state["final_response"]` 反序列化出 `ChatResponse`

### 1.3 Graph

- 当前真实工作流定义在 `app/agent/graph.py`，主链路为：
- `START -> understand -> state_update -> chatbot -> (tools?) -> observe -> respond -> memory_update -> response -> memory_compact -> END`

- 各节点职责如下：
- `understand`
  - 调用 `BasicTools.detect_object()` 和 `BasicTools.perceive_signals()`
  - 做轻量主题猜测和输入信号判断
- `state_update`
  - 读取 `read_memory_bundle`
  - 恢复 `history`、`last_agent_question`、`pending_topic`、`current_topic`
- `chatbot`
  - 调用 `ModelService.reason_next_action()`
  - 在 `direct / retrieve_knowledge / read_memory` 中做一次受控 ReAct 决策
- `tools`
  - 使用 LangGraph `ToolNode`
  - 执行 `BasicTools.as_langgraph_tools()` 暴露的工具
- `observe`
  - 从 `ToolMessage` 中解析工具输出
  - 回填 `retrieved_chunks`、`short_Memory`、`Memory`、`tool_success`
- `respond`
  - 调用 `ModelService.generate_final_response()`
  - 生成 `message_draft`、`follow_up_question`、`confidence` 等
- `memory_update`
  - 每轮默认写入 `working` 和记一条拼接后的 `episodic`
- `response`
  - 把 state 组装成标准 `ChatResponse`
- `memory_compact`
  - 以 10 条为一批压缩长期记忆

### 1.4 Memory

- Memory 总入口是 `app/memory/manager.py` + `app/memory/tool.py`。
- 当前记忆分四类：
  - `working`
    - 会话内短期记忆
    - 保存在进程内存里，带 TTL 和容量控制
    - 不落盘，服务重启后丢失
  - `episodic`
    - 长期事件记忆
    - SQLite 持久化 + 本地向量索引
  - `semantic`
    - 长期抽象语义记忆
    - SQLite 持久化 + 本地向量索引
  - `perceptual`
    - 为图片/多模态能力预留的长期记忆类型
    - 当前主链路默认不写入
- 当前读写特点：
  - `state_update` 每轮开头固定读 session/profile bundle
  - `memory_update` 每轮结束默认写 working + episodic
  - `get_profile_snapshot()` 当前只是把长期记忆内容拼成 `memory_summaries`
  - `memory_reset_on_start=True` 默认会删除 memory DB，因此长期记忆默认不稳定持久

### 1.5 RAG

- RAG 入口是 `app/rag/retriever.py`，由 `LocalKnowledgeRetriever` 实现。
- 数据源来自 `KG/` 目录下的 `.txt` 和 `.pdf`。
- 首次启动会读取语料、切块、调用 embedding 接口并构建 FAISS 索引。
- 检索工具通过 `BasicTools._retrieve_knowledge()` 暴露给 graph。
- 当前边界：
  - embedding 配置不可用时，本地检索会退化为空结果
  - RAG 更适合稳定知识，不负责时间敏感信息
  - 时间敏感问题只能再由 planner 选择 `tavily_search`

### 1.6 Tools

- 当前 graph 真正接入的工具只有 3 个：
  - `retrieve_knowledge`
  - `tavily_search`
  - `read_memory_bundle`
- `BasicTools` 同时承担两类非工具职责：
  - 启发式 topic/object 检测
  - 输入信号抽取

### 1.7 图片输入链路现状

- 当前系统已经支持 `image_base64` 和 `image_url` 两种图片输入方式。
- `reason_next_action()` 阶段会把原图传给规划模型。
- `generate_final_response()` 阶段也会把原图继续传给回复模型。
- 这意味着同一张图在不同用户问题下，可以重新基于原图进行多模态理解，而不是依赖预先生成的图片摘要。

## 2. 当前缺口

### 2.1 流式输出缺口

- 当前接口是一次性 JSON 返回。
- 前端 `fetch -> response.json()` 后才开始渲染 assistant 回复。
- 用户等待期间只能看到“思考中”，不能边生成边展示回答。

### 2.2 图片原图透传说明不够清晰

- 当前系统已经支持把 `image_base64 / image_url` 直接传给模型。
- 但 v1.3 里没有把这条能力明确写成“原图本身继续作为上下文”的设计说明。
- v1.4 不需要新增 `summary / objects / scene / ocr_text` 这类图片摘要字段。
- 更合适的目标是保持原图继续进入模型链路，便于用户围绕同一张图在不同 prompt 下重新提问。

### 2.3 一句话总结

- v1.3 当前最明显的体验缺口，不在“能不能接图”，而在“前端不能流式展示回答”，以及“图片能力缺少明确的原图透传说明”。

## 3. v1.4 目标与 backlog

### 3.1 P0: `response` 节点改为 SSE 流式输出到前端

- 目标范围锁定为“只流回答文本”。
- 不设计复杂事件流，不要求流 tool 状态、reason 或中间节点事件。
- 后端提供文本增量输出，前端逐段渲染 assistant 消息。
- 需要一个明确的结束信号，便于前端收尾。

### 3.2 P1: 图片输入保持原图透传

- 明确不做图片 `summary / objects / scene / ocr_text`。
- 目标是让原图继续作为多模态输入进入模型，而不是先转成摘要文本。
- 这样同一张图在不同用户问题下，可以重新让模型基于原图理解，而不是受固定摘要限制。
- 这不是长期图片记忆，也不是会话级图片复用，只是当前轮的原图透传策略说明。
- 只补与 `SSE` 和原图透传直接相关的测试。
- 不扩展到长期图片记忆、图片摘要抽取、图像 OCR 结构化等更大改造。

## 4. 需要补充的测试

### 4.1 SSE 测试

- 流式接口能返回 `text/event-stream`
- 回答文本可以按增量顺序输出
- 前端能正确拼接多段回答
- 流结束后不会重复渲染或丢失尾段

### 4.2 图片原图透传测试

- 图片 URL 输入时，`reason` 和 `respond` 阶段都能继续拿到原图
- Base64 图片输入时，`reason` 和 `respond` 阶段都能继续拿到原图
- 图文混合输入时，原图透传不影响文本问题本身
- 无图片输入时，纯文本链路不受影响

### 4.3 集成测试

- `SSE + 图片输入` 同时存在时，前端能边接收回答边保留当前轮图片上下文
- 同一张图在不同问题下重复请求时，后端不会因为固定摘要而限制模型理解

### 4.4 回归测试

- 纯文本请求仍可正常返回
- 非流式兼容路径若保留，响应结构不被破坏

## 5. v1.4 backlog 草案

1. `SSE Chat Streaming`
   后端支持 `text/event-stream`，前端改为增量渲染回答文本。
2. `Original Image Pass-through`
   明确并保持原图透传策略，不引入图片摘要字段。

## 结论

- v1.3 已经具备可运行的受控 ReAct 工作流、Memory、RAG、Tools 和多模态输入能力。
- 对 v1.4 而言，当前最值得优先推进的不是图片摘要或长期图片记忆，而是：
  - 让回答能以 `SSE` 方式流式展示到前端
  - 明确并保持图片输入的原图透传策略
- 只要这两条链路稳定下来，后续再考虑更复杂的多模态记忆设计会更稳妥。
