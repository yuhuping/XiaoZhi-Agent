# v1.6 Dual Framework Design: Plan-and-Execute + Multi-Round ReAct

## Overview

v1.6 refactors the agent pipeline from a single-action DAG into two distinct reasoning frameworks dispatched by interaction mode:

- **Education mode** → Plan-and-Execute subgraph (planner + executor, optimized for step-by-step problem solving)
- **Parent / Companion mode** → Multi-round ReAct subgraph (thought-action-observation loop, max 3 iterations)

The top-level Router Graph shares common nodes (understand, state_update, memory_update, response, memory_compact) and dispatches to the appropriate subgraph via `interaction_mode`.

## Architecture

### Top-Level Router Graph

```
understand → state_update → route_by_mode ─┬→ plan_execute subgraph ─┬→ memory_update → response → memory_compact → END
                                            └→ react subgraph ───────┘
```

```python
def route_by_mode(state: AgentState) -> str:
    if state.get("interaction_mode") == "education":
        return "plan_execute"
    return "react"
```

### PlanExecute Subgraph (education mode)

```
plan → plan_route ─┬→ tools → observe → execute → END
                   └→ execute → END
```

- **PlanNode** (`app/agent/nodes/plan.py`): Calls LLM once to generate internal step list + retrieval decision. Outputs JSON `{"steps": [...], "needs_retrieval": bool, "retrieval_query": "..."}`. On JSON parse failure, falls back to single-step `["directly answer the question"]`.
- **plan_route**: Conditional edge. If `needs_retrieval` is true (and tool_calls constructed by PlanNode), routes to tools; otherwise skips to execute.
- **tools → observe**: Reuses existing `ToolNode` and `ObserveNode`. Populates `retrieved_chunks` in state.
- **ExecuteNode** (`app/agent/nodes/execute.py`): Takes `plan_steps` + optional `retrieved_chunks`, calls LLM with streaming (`on_delta` callback) to generate the full solution following the planned steps. Writes `execution_result` to state.

**LLM calls**: Maximum 2 (plan + execute). No tool-calling LLM overhead.

**PlanNode prompt design**:
- Requires JSON output with steps (max 5), retrieval flag, and query
- Injects `interaction_mode`, `user_input`, memory context
- Simple questions (chat, basic arithmetic) get 1-step plans

**ExecuteNode prompt design**:
- System prompt instructs step-by-step explanation with clear markers ("Step 1: ...")
- Injects plan steps, retrieved chunks (if any), and memory context
- Education-mode tone: guiding, encouraging, pedagogical

### ReAct Subgraph (parent / companion mode)

```
reason ─┬→ tools → observe → should_continue ─┬→ reason (next iteration)
        └→ respond (direct, no tool needed)    └→ respond (terminate)

respond → END
```

- **ReasonNode**: Existing `ReasonNode` with minor additions — writes `react_iteration` and appends to `react_history` each round.
- **tools → observe**: Existing `ToolNode` + `ObserveNode`. ObserveNode appends observation to `react_history[-1]`.
- **should_continue_react**: Routing function (not a node). Terminates the loop when:
  1. `react_iteration >= react_max_iterations` (default 3)
  2. `selected_act == "direct"` (reason decided no more tools needed)
- **RespondNode**: Existing `RespondNode`. Prompt now includes `react_history` so the final answer synthesizes all iterations.

**Iteration tracking**:
```python
react_history = [
    {"iteration": 1, "thought": "...", "action": "tavily_search", "action_input": {...}, "observation": "..."},
    {"iteration": 2, "thought": "...", "action": "tavily_search", "action_input": {...}, "observation": "..."},
]
```

**Skill compatibility**: Skills (e.g., `generate_parent_summary`) work unchanged — reason selects `act="skill"` → tools executes → observe processes → respond generates.

## AgentState Changes

### Fields to Delete
- `normalized_text` — never read by any node
- `source_mode` — always `"llm"`, hardcode in `ChatMetadata`/`ReactInfo` instead
- `detected_object` — duplicate of `topic_hint`, unused

### Fields to Add
| Field | Type | Default | Used By |
|-------|------|---------|---------|
| `plan_steps` | `list[str]` | `[]` | PlanNode → ExecuteNode |
| `plan_raw` | `str` | `""` | PlanNode (raw LLM output for debugging) |
| `execution_result` | `str` | `""` | ExecuteNode (complete solution text) |
| `react_iteration` | `int` | `0` | ReasonNode (incremented each round) |
| `react_max_iterations` | `int` | `3` | should_continue_react |
| `react_history` | `list[dict]` | `[]` | ReasonNode + ObserveNode + RespondNode |

### source_mode Removal
- `ChatMetadata.source_mode`: hardcode `"llm"`
- `ReactInfo.source_mode`: hardcode `"llm"`
- Remove from `ResponseDraft` and `ReasonDecision`

## Graph Assembly (graph.py)

```python
class AgentGraph:
    def __init__(self, ...):
        plan_execute_graph = self._build_plan_execute_subgraph()
        react_graph = self._build_react_subgraph()

        router = StateGraph(AgentState)
        router.add_node("understand", UnderstandNode(...))
        router.add_node("state_update", StateUpdateNode(...))
        router.add_node("plan_execute", plan_execute_graph)
        router.add_node("react", react_graph)
        router.add_node("memory_update", MemoryUpdateNode(...))
        router.add_node("response", ResponseNode(...))
        router.add_node("memory_compact", MemoryCompactNode(...))

        router.set_entry_point("understand")
        router.add_edge("understand", "state_update")
        router.add_conditional_edges("state_update", route_by_mode, {...})
        router.add_edge("plan_execute", "memory_update")
        router.add_edge("react", "memory_update")
        router.add_edge("memory_update", "response")
        router.add_edge("response", "memory_compact")
        router.add_edge("memory_compact", END)
```

## Testing Strategy

### New Test Files

| File | Coverage |
|------|----------|
| `tests/test_plan_execute.py` | PlanNode, ExecuteNode, plan_route |
| `tests/test_react_loop.py` | Multi-round ReAct cycling |
| `tests/test_graph_routing.py` | Top-level route_by_mode dispatch |

### Test Cases

**test_plan_execute.py** (4 cases):
- Plan generates valid JSON → `plan_steps` populated in state
- Plan with `needs_retrieval=true` → tools → observe → execute path
- Plan with `needs_retrieval=false` → direct to execute path
- Plan JSON parse failure → fallback single-step degradation

**test_react_loop.py** (4 cases):
- Single-round (reason selects direct) → straight to respond
- Multi-round (2 tool calls then respond) → `react_iteration` and `react_history` accumulate correctly
- Hit `max_iterations` → forced termination to respond
- Skill route within ReAct → works unchanged

**test_graph_routing.py** (2 cases):
- `interaction_mode="education"` → plan_execute subgraph
- `interaction_mode="parent"` or `"companion"` → react subgraph

### Test Approach
- Mock `ModelService` LLM calls (`generate_plan`, `execute_plan`, `reason_next_action`, `generate_final_response`)
- Use mock tool functions for ToolNode
- Verify state field changes and node execution order via `workflow_trace`
- Existing tests (SSE streaming, memory_update, etc.) should pass unchanged — they exercise parent/companion mode which maps to ReAct subgraph with single-iteration behavior identical to current pipeline

## Edge Cases

| Scenario | Handling |
|----------|----------|
| Plan JSON parse error | Fallback to `["directly answer"]`, single-step execute |
| Plan has only 1 step | Normal execute, works for simple questions / chat in education mode |
| ReAct all iterations exhaust tools | Force respond after max_iterations with whatever info gathered |
| ReAct first round is direct | Equivalent to current single-action behavior |
| Education mode casual chat | Planner generates 1-step plan, execute responds normally |
