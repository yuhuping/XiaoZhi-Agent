# XiaoZhi v1.4 规格

## 背景

- v1.4 的范围直接来源于 [v1.3 审计](/home/yhp/projects/XiaoZ/docs/v1.3-audit.md)，目标是把已经确认的问题整理成下一步规格，而不是重新展开一份全量 roadmap。
- 当前版本已经具备可运行的核心能力：
  - 基于 ReAct 的受控推理链路
  - 基于 LangGraph 的工作流编排
  - 会话与长期 Memory 机制
  - 本地 RAG 检索能力
  - `image_base64` / `image_url` 图片输入能力
- v1.4 只聚焦两类缺口：
  - 回答无法以 `SSE` 方式流式展示到前端
  - 图片输入虽然已可用，但“原图继续作为上下文透传”的设计说明还不够明确

## v1.3 暴露出的缺点

### 1. 当前回答仍是一次性返回

- 当前主接口 `POST /api/v1/chat/explain-and-ask` 返回的是完整 `ChatResponse` JSON。
- 前端目前采用 `fetch -> response.json()` 的消费方式，只有在服务端生成完整回答后才开始渲染。
- 用户等待期间只能看到“思考中”状态，无法边生成边查看回答内容。

### 2. 图片原图透传策略说明不够清晰

- 当前系统已经支持 `image_base64` 与 `image_url` 两种图片输入方式。
- 当前实现中，原图已经会进入 `reason_next_action()` 与 `generate_final_response()` 两个阶段。
- 但 v1.3 没有把这件事明确写成规格，因此外部读者容易误以为系统需要先做图片摘要，或者后续要补 `summary / objects / scene / ocr_text` 之类的字段。

### 3. 一句话总结

- v1.4 的重点不是增强图片摘要能力，而是补齐用户体验与规格表达，把“流式回答”和“原图透传”这两件已经明确的方向定清楚。

## v1.4 功能规格

### 1. SSE 流式回答

- v1.4 主对话接口统一改为 `text/event-stream`，不再把现有一次性 JSON 返回作为主兼容路径。
- 流式范围只包含 assistant 的最终回答文本，不扩展为多事件类型协议。
- v1.4 不要求流出以下内容：
  - tool 执行状态
  - reason 文本
  - 中间节点事件
  - 调试信息
- 前端需要把多个文本增量拼接为同一条 assistant 消息，而不是把每个 chunk 渲染成一条独立消息。
- 流式协议必须提供明确结束信号，用于：
  - 停止 loading 状态
  - 收尾 UI
  - 避免尾段丢失
  - 避免最后一段被重复渲染
- 若前端仍需消费最终元数据，应在流结束阶段一次性交付，不把当前 `ChatResponse` 结构拆解成复杂事件总线。

### 2. 图片原图透传

- v1.4 保持现有输入边界不变，继续使用：
  - `image_base64`
  - `image_url`
  - `image_mime_type`
- 原图在当前轮请求中需要持续透传到以下两个关键阶段：
  - `reason_next_action()`
  - `generate_final_response()`
- 同一张图在不同问题下，应允许模型重新基于原图进行理解，而不是依赖某个预先生成的固定摘要。
- v1.4 明确不引入以下字段或机制：
  - `summary`
  - `objects`
  - `scene`
  - `ocr_text`
  - 长期图片记忆
  - 图片结构化抽取
- 原图透传是“当前轮多模态输入策略”的说明，不等同于长期 perceptual memory，也不等同于跨轮图片复用能力。

### 3. 非目标

- 不扩展长期 perceptual memory。
- 不设计多事件类型 SSE 协议。
- 不做图片 OCR 结构化、场景解析或摘要缓存。
- 不在 v1.4 中引入与 v1.3 审计无关的大范围链路改造。

## 接口与类型影响

### 1. 对外接口

- `POST /api/v1/chat/explain-and-ask` 的响应介质在 v1.4 中调整为 `text/event-stream`。
- 请求体字段继续沿用现有 `ChatRequest` 输入面：
  - `text`
  - `image_base64`
  - `image_url`
  - `image_mime_type`
  - `age_hint`
  - `session_id`
  - `mode`
  - `profile_id`
- v1.4 不新增任何图片摘要类输入字段，也不新增额外的前端图片上传字段。

### 2. 前端消费方式

- 前端需要从一次性 `fetch -> response.json()` 调整为流式消费与增量渲染。
- 前端对 assistant 回复的渲染语义改为：
  - 接收文本增量
  - 追加到当前消息
  - 收到结束信号后停止等待并完成收尾
- 响应语义不再是“一次性完整 JSON”，而是“文本流 + 明确结束信号”。

## 测试与验收

### 1. SSE 验收

- 流式接口返回头为 `text/event-stream`。
- 回答文本按顺序增量到达。
- 前端能把多段文本正确拼接成单条 assistant 回复。
- 流结束后前端能正常收尾，不会重复渲染，也不会丢失尾段。

### 2. 图片原图透传验收

- `image_url` 输入时，`reason` 和 `respond` 阶段都能继续访问原图。
- `image_base64` 输入时，`reason` 和 `respond` 阶段都能继续访问原图。
- 图文混合输入时，原图透传不影响文本问题本身的理解。
- 无图片输入时，纯文本链路不受影响。

### 3. 联合场景验收

- `SSE + 图片输入` 同时存在时，前端能边接收回答边保留当前轮图片上下文。
- 同一张图搭配不同文本问题重复请求时，规格上仍以原图作为理解依据，而不是固定摘要。

## backlog 对应关系

- `SSE Chat Streaming`
  - 对应 v1.4 中“`SSE` 流式回答”规格。
- `Original Image Pass-through`
  - 对应 v1.4 中“图片原图透传”规格。

## 结论

- v1.4 不追求在图片能力上继续横向扩张，而是先把用户最直接能感知到的体验缺口补齐。
- 只要流式回答与原图透传策略在规格、接口与测试层面被明确下来，后续再讨论更复杂的多模态记忆或图片结构化能力会更稳妥。
